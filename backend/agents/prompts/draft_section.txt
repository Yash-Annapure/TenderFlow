SECTION-TYPE RULES

── EXECUTIVE SUMMARY ──────────────────────────────────────────
• Paragraph 1: what Meridian proposes (specific deliverable and dataset scope),
  grounded in the tender's stated objective.
• Paragraph 2: the core output — size of dataset, key indicators, delivery timeline
  drawn from the tender's duration and deliverables section.
• No bullets. 2 tight paragraphs only.

── 1. PROBLEM FRAMING ─────────────────────────────────────────
• Name the specific regulatory or policy gap described in the tender's background section.
• Explain why existing data sources or approaches are insufficient (use the tender's
  own language as a reference; do not copy verbatim).
• State Meridian's specific angle — how our method addresses the gap the tender
  identifies.
• 3 paragraphs maximum. No bullets.

── 2. ENTITY TYPOLOGY ─────────────────────────────────────────
• OUTPUT FORMAT: a markdown pipe table.
• Columns: Entity/Provider Type | Relevance to Contract | Classification Approach
• Rows: one per entity category mentioned in the tender's scope section.
  IMPORTANT: derive the categories directly from the tender's scope/objectives text.
  Do NOT use generic categories — use what the tender specifies.
• Minimum 5 rows, maximum 8.
• Relevance column: "High", "Medium", or "Low" with a one-line justification.
• Classification Approach column: be specific (name the data signals, registries,
  or methods used to identify this entity type).

── 3. METHODOLOGY ─────────────────────────────────────────────
• Numbered subsections: 3.1 Population Identification, 3.2 Classification and Profiling,
  3.3 Monitoring and Quality Assurance.
• Each subsection: 3-5 sentences. Be specific — name data sources, precision targets,
  update frequencies, and QA steps the tender requires.
• Where the tender specifies a methodological requirement (e.g. evidence-based
  classification, false-positive rejection), address it explicitly by name.
• Pull specific figures and methods from KB evidence where available.

── 4. DELIVERABLES ────────────────────────────────────────────
• OUTPUT FORMAT: a markdown pipe table.
• Columns: Deliverable | Description | Timeline
• CRITICAL: extract the deliverables DIRECTLY from the tender document's deliverables
  section. Preserve the tender's own deliverable labels (D1, D2, etc.) and timelines
  exactly as specified. Do not invent deliverables not listed in the tender.
• Description column: write one concise sentence summarising the deliverable, using
  the tender's description but in Meridian's voice.
• Include ALL deliverables listed in the tender. If the tender lists D1-D7, produce
  all 7 rows.

── 5. TEAM ────────────────────────────────────────────────────
• OUTPUT FORMAT: a markdown pipe table.
• Columns: Name | Role | Input (Days)
• Map team members to the key expertise requirements listed in the tender's team
  section. Each required expertise must be covered by at least one team member.
• Use REAL names from CV evidence if available in the knowledge base.
  If a CV name is available, use it — do not use placeholders like "TBC".
• Day counts should be proportional to contract duration and deliverable complexity.
  For a 24-month contract at EUR 3M budget, expect 300-600 total person-days.
• End with a Total row: | **Total** | | **<sum>** |

── 6. PRICE ───────────────────────────────────────────────────
• OUTPUT FORMAT: a markdown pipe table.
• Columns: Cost Category | Amount (EUR)
• Anchor the total to the tender's estimated contract value if stated.
• Line items: Staff costs, Infrastructure and data sources, Subcontracting (if relevant),
  Travel and accommodation, Contingency (5%), TOTAL (excl. VAT).
• Staff costs = sum of (person-days × day rate). Use €850/day for senior profiles,
  €700/day for technical, €600/day for analyst. Make internally consistent with Team table.
• Bold the TOTAL row: | **TOTAL (excl. VAT)** | **€X,XXX,XXX** |
